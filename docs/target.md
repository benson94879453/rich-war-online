# 1. 遊戲核心定位
Rich War Online 是一款以大富翁資產經營為基礎，加入「干預卡」與「干預窗口」機制的線上多人桌遊。
玩家不只在自己的回合操作，也能在他人回合中透過卡牌影響骰點、阻止交易、保護資產或製造突發變局。
## 核心賣點
傳統大富翁的主要問題是「非自己回合等待感太重」。
本作的核心解法是：
> 讓玩家在別人的回合也有機會出手，形成全員參與的動態博弈。
MVP 首要驗證問題：
> 「大富翁 + 他人回合插手」是否真的好玩？

# 3. 回合結構
每名玩家的回合分為：
1. 擲骰相 Dice Phase
2. 行動相 Action Phase
3. 回合結束 Turn End
---
## 3.1 擲骰相
流程：
1. 開啟【干預窗口 1：擲骰前】
2. 主動玩家擲骰
3. 套用狀態效果，例如加速、緩速
4. 棋子逐格移動
5. 抵達目標格
6. 開啟【干預窗口 2：格子效果生效前】
7. 結算格子效果
---
## 3.2 行動相
主動玩家可依格子與狀態執行：
* 購買地產
* 升級建物
* 出售 / 管理建物
* 股市買賣
* 主動棄牌
* 使用允許在此時機使用的干預卡
一般主動操作不開啟完整干預窗口，避免節奏過慢。
但若該操作可被特定卡牌回應，系統會開啟「指定反應窗口」。
---
## 3.3 回合結束
流程：
1. 主動玩家宣告結束
2. 開啟【干預窗口 3：回合結束前】
3. 無人響應或超時
4. 清理暫時狀態
5. 換下一位玩家
---
# 4. 干預窗口規則
MVP 僅保留三個主要干預窗口：
| 窗口     | 時機            | 主要用途          |
| ------ | ------------- | ------------- |
| 干預窗口 1 | 擲骰前           | 改骰、重骰、影響移動    |
| 干預窗口 2 | 抵達格子後，格子效果生效前 | 免租、移位、反制事件    |
| 干預窗口 3 | 回合結束前         | 最後出手、保護、偷竊、干擾 |
## 4.1 自動跳過條件
若所有玩家在目前窗口中：
* 沒有可用干預卡，或
* 已選擇不響應
則系統自動跳過等待時間。
## 4.2 指定反應窗口
部分行動不會開啟完整干預窗口，但可觸發短暫指定反應窗口。
例如：
| 行動     | 可觸發反應 |
| ------ | ----- |
| 宣告購買地產 | 打斷購買  |
| 建物被攻擊  | 建物護盾  |
| 干預卡被打出 | 反制卡   |
| 事件即將生效 | 事件反制  |
指定反應窗口只允許使用對應類型的卡，不開放所有通用干預。

# 5. 干預卡系統
## 5.1 卡牌分類
| 分類   | 說明                       |
| ---- | ------------------------ |
| 擲骰干預 | 主要用於干預窗口 1               |
| 行動干預 | 用於特定行動或指定反應窗口            |
| 通用干預 | 可在多數干預窗口使用               |
| 反制卡  | 干預卡特殊子類，只能回應其他卡牌、事件或指定行動 |
## 5.2 MVP 反制規則
* 每個干預窗口內，每位玩家最多打出 1 張干預卡
* 干預卡打出後，其餘玩家依座位順時針順序各有一次反制機會
* 反制成功則原卡無效
* MVP 不開啟第二層反制
* 打出反制後，該玩家本窗口不能再打其他干預卡
* 若多人同時提交，以 Host 收到時間 + 座位順序決定優先
## 5.3 手牌規則
| 項目   | 設定               |
| ---- | ---------------- |
| 起始手牌 | 2 張              |
| 手牌上限 | 4 張              |
| 補牌方式 | 踩到補牌格抽 1 張干預卡    |
| 滿手補牌 | 抽 1 張後棄 1 張      |
| 主動棄牌 | 行動相可棄任意張         |
| 牌庫循環 | 干預卡用完進棄牌堆，牌庫空時洗回 |
| 牌庫分離 | 干預卡、機會牌、命運牌三庫獨立  |
---
# 6. 棋盤格子系統
MVP 棋盤約 40 格，採資料驅動設計。
## 6.1 格子類型
| 格子類型     | 功能          |
| -------- | ----------- |
| 起點格      | 經過或停留時領薪水   |
| 房地產格     | 可購買、升級、收租   |
| 股市格      | 可買賣股票       |
| 補牌格      | 抽 1 張干預卡    |
| 機會格      | 抽機會事件牌並立即結算 |
| 命運格      | 抽命運事件牌並立即結算 |
| 祝福 / 詛咒格 | 賦予狀態        |
| 特殊地點格    | 醫院、監獄等      |
| 空白 / 安全格 | 無效果，用作節奏緩衝  |
---
# 7. 玩家狀態系統
## 7.1 MVP 狀態
| 狀態   | 效果             | 持續      |
| ---- | -------------- | ------- |
| 加速   | 擲骰點數 ×2        | 1 回合    |
| 緩速   | 擲骰點數 ×0.5，最低 1 | 1 回合    |
| 幸運   | 負面格子無效化一次      | 2 回合    |
| 霉運   | 過路費增加 50%      | 2 回合    |
| 住院中  | 跳過擲骰與主動行動      | 依事件     |
| 入獄中  | 跳過擲骰，可付保釋金     | 最多 3 回合 |
| 建物護盾 | 己方建物免疫攻擊       | 1 回合    |
## 7.2 狀態邊界規則
* 加速與緩速同時存在時互相抵消，視為 ×1
* 緩速後最低移動 1 格，0 步不合法
* 住院 / 入獄中仍可收租
* 住院 / 入獄中仍會被扣款、被事件影響
* 住院 / 入獄中不能主動打行動干預卡
* 住院 / 入獄中仍可打通用干預卡
---
# 8. 勝利條件
MVP 採限時 + 輪數雙觸發模式。
## 8.1 結算條件
以下任一條件達成時，遊戲結束並進入資產結算：
* 遊戲時間達 45 分鐘
* 達成 30 輪
一輪定義為：所有未破產玩家各完成一次自己的回合。
## 8.2 資產計算
總資產 =
> 現金 + 地產估值 + 股票市值
同分時，以現金較多者獲勝。
## 8.3 破產規則
玩家破產後：
* 變成破產旁觀者
* 可留在房間內觀看
* 不能再打干預卡
* 不支援局外玩家中途加入觀戰
---
# 9. 股市系統，MVP 極簡版
股市保留，但降權為輔助財富系統。
## 9.1 MVP 規格
* 3 支股票
* 穩健型 / 成長型 / 投機型
* 每回合更新一次價格
* 使用簡單隨機遊走
* 僅踩到股市格可交易
* 不做圖表 UI，只顯示數字清單
* 不做崩盤 / 狂牛行情
## 9.2 設計目的
股市不是主要勝利路線。
保留股市的主要原因是讓「股市操控」類干預卡有作用場景。

# 10. 技術架構
## 10.1 Godot 版本
開發鎖定：
> Godot 4.6.x stable
不追逐 RC 或預發布版本，避免 Web 匯出與多人同步風險。
## 10.2 目標平台
| 平台      | 優先度 |
| ------- | --- |
| Web     | 最高  |
| Windows | 次要  |
| macOS   | 次要  |
## 10.3 多人同步策略
MVP Web 多人主線：
> WebSocketMultiplayerPeer
桌面版可選：
> ENetMultiplayerPeer
但 ENet 不作為 Web 備選方案。
## 10.4 權威模型
採 Host 權威模型：
* Client 只送出 intent
* Host 驗證行動是否合法
* Host 計算結果
* Host 廣播結果
* Client 只負責顯示與輸入

---
## P0.1 本地棋盤原型
目標：
> 在單機環境中，讓 2 名玩家輪流擲骰並移動。
完成條件：
* Godot 專案建立
* GameManager 設為 Autoload
* 生成 40 格假棋盤
* 建立 2 名測試玩家
* 按鈕擲骰
* 棋子移動
* 顯示目前玩家
* 顯示骰點
* 回合自動切換
不做：
* 買地
* 干預卡
* 股市
* 多人
* 美術
* 正式 UI
---
## P0.2 WebSocket 同步原型
目標：
> 兩個瀏覽器 / 視窗連到同一局，Host 擲骰後雙方同步結果。
完成條件：
* Host 可建立房間
* Client 可加入房間
* Client 發送擲骰請求
* Host 驗證是否為當前玩家
* Host 產生骰點
* Host 更新玩家位置
* Host 廣播骰點與新位置
* 所有 Client 同步棋子移動
此階段只同步一件事：
> 擲骰結果與棋子位置
---
## P0.3 最小 GameState 快照
目標：
> 建立可序列化的 GameState，為斷線重連與同步打基礎。
GameState 至少包含：
* 玩家列表
* 目前玩家 index
* 玩家位置
* 玩家現金
* 棋盤基本資料
* 回合數
* 隨機種子或最近一次骰點
* 遊戲狀態
完成條件：
* 可將目前遊戲狀態轉成 Dictionary / JSON
* 可從 Dictionary / JSON 還原遊戲狀態
* Client 加入時可同步完整狀態
